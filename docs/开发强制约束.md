# 开发强制约束

> 本文档是 AI 和开发者在本仓库内改代码时必须遵守的硬性约束。目标是减少随意生成、风格漂移、重复造轮子和破坏既有架构。

## 适用范围

- 适用于 `src/`、`mock/`、`types/`、`docs/` 下的所有新增和修改。
- 适用于页面、组件、Hook、API、Store、路由、样式、Mock、类型声明、文档示例。
- 当用户需求、本文档、工程化规范、组件文档、现有代码存在冲突时，优先级为：用户明确要求 > 本文档 > `docs/工程化规范.md` > `docs/组件使用规范/*` > 现有局部代码风格。

## AI 开发前必须执行

1. 先阅读相关文件，不允许只凭文件名猜实现。
2. 先确认项目现有能力：组件、Hook、Store、工具函数、样式变量、类型定义。
3. 涉及表单、表格、搜索、弹窗时，必须先参考 `docs/组件使用规范/`。
4. 涉及架构、目录、命名、提交、质量检查时，必须先参考 `docs/工程化规范.md`。
5. 修改前检查当前工作区是否已有改动；不得回滚、覆盖、格式化无关文件。

## 技术栈不可变更

- 必须使用 Vue 3 + TypeScript + Vite + Element Plus。
- 状态管理必须使用 Pinia，不新增 Vuex、Redux 等状态库。
- 路由必须使用 Vue Router 4，保持 Hash 路由模式。
- 样式优先使用项目现有 SCSS、CSS 变量、UnoCSS、Element Plus 变量体系。
- HTTP 请求必须走 `src/axios` 封装，不允许在业务代码里直接新建 axios 实例或直接 `fetch` 调接口。
- 日期处理优先使用项目已有 `dayjs` 或 `src/utils/dateUtil.ts`。
- 工具函数优先复用 `src/utils/`、`lodash-es`、`@vueuse/core`，不复制粘贴相似实现。

## 依赖约束

- 禁止为单个小需求随意新增依赖。
- 新增依赖前必须说明原因、替代方案、体积和维护风险。
- 不允许新增与现有能力重复的 UI 库、状态库、请求库、日期库、图标库。
- 包管理器固定使用 `pnpm`；不新增 `yarn.lock`、`pnpm-lock.yaml` 等额外锁文件，也不得无故重写现有锁文件。

## 代码风格硬约束

- Vue 单文件组件优先使用 `<script setup lang="ts">`。
- TSX 仅在复杂渲染配置、组件内部渲染函数或现有文件已使用 TSX 时使用。
- 引入路径优先使用 `@/` 别名，不使用过深的 `../../../`。
- 函数、变量使用清晰业务命名，不使用 `data1`、`list2`、`temp` 这类无意义命名。
- 不允许新增大段无类型的 `any`；确实无法避免时必须限制作用域。
- 不允许把接口返回值、表单值、表格行数据长期保持为隐式结构；需要定义对应类型。
- 编写方法、功能函数或复杂逻辑时，必须添加必要注释，说明业务意图、关键分支或非显而易见的实现原因。
- 不允许在组件中写无意义注释；只为复杂业务规则、兼容逻辑、非显而易见的实现补充注释。
- 不允许提交 `console.log`、`debugger`、临时注释代码、无用 import、未使用变量。

## Vue 组件约束

- 页面组件放在 `src/views/业务模块/` 下，局部组件放在页面目录的 `components/` 下。
- 可复用通用组件放在 `src/components/组件名/`，并通过 `index.ts` 导出。
- 组件 props、emits 必须声明清楚；复杂 props 必须有类型。
- 组件对外暴露方法时使用 `defineExpose` 或项目已有 register 模式，避免父组件直接依赖内部 DOM。
- 修改 `docs/组件使用规范/` 已覆盖的组件时，如果 props、emits、slots、暴露方法、Hook 方法、默认行为或使用方式发生变化，必须精炼同步对应组件文档；单个组件规范文档不超过 500 行。
- 不允许把大量业务请求、表单校验、表格列配置、弹窗状态全部堆在模板里。
- 复杂页面应拆分为：API、类型、页面容器、局部组件、必要 Hook。

## 业务组件优先级

- 列表页优先使用 `Table + useTable`，参考 `docs/组件使用规范/Table表格组件文档.md`。
- 查询区优先使用 `Search + useSearch`，参考 `docs/组件使用规范/Search搜索组件.md`。
- 表单优先使用 `Form + useForm` 或 schema 配置，参考 `docs/组件使用规范/Form组件使用文档.md`。
- 弹窗优先使用 `Modal`，参考 `docs/组件使用规范/Modal弹窗组件.md`。
- 基础按钮优先使用全局 `BaseButton` 或 Element Plus 按钮，保持项目视觉一致。

## API 与数据约束

- API 文件放在 `src/api/模块名/index.ts`。
- API 类型放在同目录 `types.ts`，或在模块内拆分更细类型文件。
- API 方法命名使用动词 + 资源 + `Api`，例如 `getUserListApi`、`createUserApi`、`updateUserApi`、`deleteUserApi`。
- 所有 API 方法必须声明返回 Promise 类型，例如 `Promise<IResponse<UserListRes>>`。
- 请求路径、参数名、返回结构必须以需求或后端约定为准，不允许 AI 自行编造字段。
- 接口异常处理优先走全局 axios 拦截器；页面只处理业务态、空态、局部提示。

## 路由与权限约束

- 静态路由维护在 `src/router/index.ts`。
- 权限相关逻辑优先查 `src/permission.ts` 和 `src/store/modules/permission.ts`。
- 路由 `name` 必须唯一，建议使用 PascalCase。
- 菜单展示相关字段放在 `meta`，如 `title`、`icon`、`hidden`、`alwaysShow`、`noTagsView`。
- 新增页面路由必须考虑 404、登录态、TagsView、菜单显示、国际化标题。

## Store 约束

- 全局状态使用 `src/store/modules/` 下 Pinia 模块。
- Store 只放跨页面共享、需要持久化或与权限/用户/布局强相关的数据。
- 页面私有状态不要放进全局 Store。
- setup 外调用 Store 时保持项目模式：导出并使用 `useXxxStoreWithOut()`。
- 持久化字段必须谨慎，不能把临时搜索条件、大体积列表、敏感数据随意持久化。

## 样式与 UI 约束

- 后台管理页面应保持信息密度、可扫描、克制，不做营销页式大 Hero。
- 优先复用 Element Plus、项目组件、UnoCSS 工具类和现有 SCSS 变量。
- 页面根容器优先使用 `ContentWrap` 等既有布局组件。
- 不允许硬编码大量魔法颜色；通用颜色优先使用 CSS 变量或 Element Plus 变量。
- 不允许让文本、按钮、表格列在常见宽度下溢出或重叠。
- 图标优先使用 `@element-plus/icons-vue` 或项目本地 `svg-icon` 体系。

## 国际化约束

- 已国际化的模块新增文案时，同步维护 `src/locales/zh-CN.ts` 和 `src/locales/en.ts`。
- 路由标题、菜单标题、布局级文案优先使用 i18n。
- 业务临时提示是否国际化，按当前模块已有模式保持一致。

## Mock 与环境约束

- Mock 数据放在 `mock/模块名/index.mock.ts`。
- Mock 返回结构必须尽量贴近真实接口，不能为了页面方便随意变形。
- 环境变量必须使用 `VITE_` 前缀。
- 不允许在代码中硬编码生产域名、Token、账号、密码、密钥。

## 质量检查强制项

代码改动完成后，按影响范围执行检查：

```bash
pnpm ts:check
pnpm lint:eslint
pnpm lint:style
pnpm build:dev
```

- TS 类型或共享逻辑改动：至少执行 `pnpm ts:check`。
- Vue、TS、TSX 改动：至少执行 `pnpm lint:eslint`。
- 样式改动：至少执行 `pnpm lint:style`。
- 路由、构建配置、依赖、公共组件改动：建议执行 `pnpm build:dev`。
- 如因环境、依赖、网络等原因无法执行，必须在交付说明中明确说明。

## 禁止行为

- 禁止无需求重构大面积文件。
- 禁止改动与任务无关的格式、缩进、排序。
- 禁止删除用户已有代码、注释、文档、Mock、类型，除非用户明确要求或确认为废弃。
- 禁止绕过现有封装直接调用底层库。
- 禁止复制一份几乎相同的组件或 Hook。
- 禁止把后端字段映射、权限判断、表单校验散落在多个组件中重复实现。
- 禁止为了“看起来能跑”写假字段、假权限、假流程而不标注。

## AI 交付前自检清单

交付前逐项确认：

- 需求是否完整覆盖，没有只实现 UI 或只实现数据层。
- 是否复用了现有组件、Hook、API、Store、工具函数。
- 是否新增了必要类型，避免隐式结构。
- 是否考虑加载态、空态、错误态、禁用态、分页、重置、关闭弹窗清理。
- 是否影响权限、路由、TagsView、国际化、持久化。
- 是否没有覆盖用户未请求的改动。
- 是否已执行或说明未执行质量检查。
