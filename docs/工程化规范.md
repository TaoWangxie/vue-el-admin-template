# 工程化规范

> 本文档用于指导 AI 和开发者在本项目中生成更稳定、更一致、更可维护的代码。涉及强制红线时，以 `docs/开发强制约束.md` 为准。

## 项目基线

- 技术栈：Vue 3.5、TypeScript 5.3、Vite 5、Element Plus 2.5、Pinia、Vue Router 4、Vue I18n。
- 包管理器：pnpm 8+。
- Node 版本：18+。
- 路由模式：Hash。
- 样式体系：SCSS、CSS 变量、UnoCSS、Element Plus 变量。
- 请求层：`src/axios` 基于 Axios 封装。
- Mock：`vite-plugin-mock`，目录为 `mock/`。
- 路径别名：`@/` 指向 `src/`。

## 常用命令

```bash
pnpm dev
pnpm ts:check
pnpm lint:eslint
pnpm lint:format
pnpm lint:style
pnpm build:dev
pnpm build:test
pnpm build:pro
```

## 推荐开发流程

1. 阅读需求，识别影响范围：页面、组件、API、Store、路由、样式、Mock、文档。
2. 阅读现有实现和组件文档，优先复用项目能力。
3. 先补类型和 API，再做页面状态与交互，最后处理样式细节。
4. 小步改动，避免一次性重写无关模块。
5. 完成后按影响范围执行类型检查、Lint、构建。

## 目录职责

```text
src/
  api/                 # 接口请求与接口类型
  assets/              # 图片、SVG 等静态资源
  axios/               # 请求封装、拦截器、取消请求
  components/          # 通用组件
  constants/           # 全局常量
  hooks/               # 组合式 Hook
  layout/              # 布局框架
  locales/             # 国际化语言包
  plugins/             # 插件初始化
  router/              # 路由配置
  store/               # Pinia 模块
  styles/              # 全局样式和变量
  utils/               # 通用工具函数
  views/               # 业务页面
mock/                  # Mock 接口
types/                 # 全局类型声明
docs/                  # 项目规范和组件文档
```

## 命名规范

- Vue 页面组件：`PascalCase.vue` 或按既有目录使用业务名。
- 局部组件：放入当前页面 `components/`，文件名使用 `PascalCase.vue`。
- 通用组件：`src/components/ComponentName/src/ComponentName.vue`，通过 `index.ts` 导出。
- Hook：以 `use` 开头，例如 `useTable`、`useForm`、`useUserDetail`。
- Store：`useXxxStore`，文件名使用小驼峰或业务名，例如 `user.ts`、`permission.ts`。
- API 方法：`getXxxApi`、`createXxxApi`、`updateXxxApi`、`deleteXxxApi`。
- 类型：请求参数用 `XxxParams`，响应数据用 `XxxRes`，列表项用 `XxxItem`，表单模型用 `XxxForm`。
- 常量：全大写下划线或语义化枚举，避免散落魔法字符串。

## Vue 组件规范

优先写法：

```vue
<script setup lang="ts">
import { ref } from 'vue'

defineOptions({
  name: 'ExamplePage'
})

const loading = ref(false)
</script>

<template>
  <ContentWrap title="示例">内容</ContentWrap>
</template>
```

约束：

- 页面入口保持清晰，复杂逻辑拆到局部组件或 Hook。
- props 使用类型声明，emits 声明事件名和参数。
- 表单、表格、搜索、弹窗优先使用项目封装。
- 复杂插槽渲染可使用 TSX，但不要把普通模板全部改成 TSX。
- 父子通信优先用 props/emits；跨层级共享再考虑 Store 或 mitt。

## 页面组织建议

业务页面复杂时推荐结构：

```text
src/views/system/user/
  index.vue
  components/
    UserFormModal.vue
    UserRoleDrawer.vue
  hooks/
    useUserColumns.ts
    useUserSearch.ts
```

页面容器负责：

- 组织查询、列表、弹窗和页面级交互。
- 调用 API 和 Hook。
- 处理加载态、空态、刷新、分页、删除确认。

局部组件负责：

- 独立表单、弹窗、抽屉、详情块。
- 通过 props/emits 与页面容器通信。
- 不直接修改父组件内部状态。

## API 规范

推荐结构：

```text
src/api/user/
  index.ts
  types.ts
```

示例：

```ts
import request from '@/axios'
import type { UserListParams, UserListRes } from './types'

export const getUserListApi = (params: UserListParams): Promise<IResponse<UserListRes>> => {
  return request.get({ url: '/user/list', params })
}
```

要求：

- API 层只描述请求，不写页面提示、不操作 Store、不处理 DOM。
- 参数和返回值必须有类型。
- 分页字段、状态码、业务字段以接口约定为准。
- 删除、提交、更新等操作在页面层处理二次确认和成功提示。

## Store 规范

适合放入 Store：

- 用户信息、Token、角色权限。
- 路由、菜单、TagsView。
- 布局配置、主题、语言。
- 多页面共享且生命周期长的数据。

不适合放入 Store：

- 单个页面的查询条件。
- 单次弹窗的临时表单。
- 可以由接口重新获取的大体积列表。
- 与权限无关的短生命周期 UI 状态。

## 路由规范

新增业务路由时关注：

- `path` 清晰稳定，避免后续频繁改 URL。
- `name` 全局唯一。
- `component` 使用动态导入。
- `meta.title`、`meta.icon`、`meta.hidden`、`meta.noTagsView` 按菜单和标签页需求配置。
- 需要权限时，确认当前项目使用 `server`、`frontEnd` 还是 `static` 路由策略。

示例：

```ts
{
  path: '/system/user',
  name: 'SystemUser',
  component: () => import('@/views/system/user/index.vue'),
  meta: {
    title: '用户管理',
    icon: 'ep-icon:User'
  }
}
```

## 表格规范

- 列表页优先使用 `Table + useTable`。
- 查询后回到第一页，刷新保持当前页，删除后处理页码回退。
- 金额、时间、枚举状态必须统一格式化。
- 操作列固定在右侧时注意窄屏可用性。
- 跨页选择必须配置 `row-key` 和 `reserveSelection`。
- 表格列配置建议拆到 `useXxxColumns.ts`，避免页面过长。

## 表单规范

- 表单优先使用 `Form + useForm` 和 schema。
- 必填、长度、格式、联动规则写在 schema 或专用校验函数中。
- 编辑场景打开弹窗时先重置，再回填数据。
- 关闭弹窗时清理校验和临时表单。
- 提交时避免重复点击，必须有 loading 或 disabled 保护。
- 表单字段名必须与接口字段或明确的 DTO 字段一致。

## 弹窗规范

- 弹窗新增、编辑、详情应有明确模式，例如 `create`、`edit`、`detail`。
- 标题、按钮文案、禁用态应随模式变化。
- 弹窗内部不直接假设父页面列表存在，提交成功后通过事件通知父页面刷新。
- 弹窗关闭时重置状态，避免上次数据残留。

## 搜索规范

- 搜索条件使用明确默认值。
- 重置搜索应同时重置分页到第一页。
- 远程下拉选项应处理加载态和空数据。
- 搜索项过多时优先使用项目 Search 组件的展开/收起能力。

## 样式规范

- 页面整体优先用 `ContentWrap` 承载。
- 中后台页面保持紧凑、清晰、易扫描。
- 通用间距优先使用 UnoCSS 工具类或已有组件间距。
- 局部样式使用 `scoped`，全局样式放入 `src/styles/`。
- 避免深层选择器穿透；必须穿透 Element Plus 时使用 `:deep()` 并控制范围。
- 颜色、阴影、圆角优先复用现有变量和 Element Plus 变量。
- 不写难以维护的绝对定位布局，除非是图标、浮层、特殊视觉元素。

## 图标与资源规范

- 通用图标优先使用 `@element-plus/icons-vue`，业务和品牌图标使用本地 `svg-icon`。
- 本地 SVG 放在 `src/assets/svgs`，由 `vite-plugin-svg-icons` 处理。
- 图片资源放在 `src/assets/imgs` 或业务模块资源目录。
- 不在业务代码中引用远程临时图片。

## 国际化规范

- 布局、菜单、路由、全局提示优先国际化。
- 新增语言键保持模块化命名，避免同名覆盖。
- 中文和英文语言包同步更新。
- 纯业务后台页面如当前模块未国际化，可保持模块一致，但不要混用多套风格。

## Mock 规范

- Mock 文件命名：`mock/模块名/index.mock.ts`。
- Mock 接口路径与真实接口保持一致或明确标注差异。
- Mock 数据必须覆盖列表、空数据、详情、创建、编辑、删除等关键场景。
- 随机数据可使用 `@faker-js/faker`，但字段结构必须稳定。

## 类型规范

- 业务类型就近放在 API 模块 `types.ts` 或页面模块下。
- 全局通用类型放在 `types/`。
- 避免把所有类型堆进一个巨大文件。
- 接口响应、表单模型、表格行、枚举值分开命名。
- 对可能为空的字段显式使用 `null` 或 `undefined`，不要靠运行时猜测。

## 错误与状态规范

页面至少考虑：

- 初次加载态。
- 空数据态。
- 请求失败态。
- 提交中状态。
- 删除确认和删除中状态。
- 权限不足或不可操作状态。

不要只实现理想路径。

## 提交规范

Commit message 使用 Conventional Commits，允许类型见 `commitlint.config.js`：

```text
feat: 新增用户管理页面
fix: 修复表单重置后校验未清理
docs: 补充组件使用说明
refactor: 优化表格列配置组织
types: 补充用户接口类型
```

## 质量门禁

按改动范围选择命令：

```bash
pnpm ts:check       # 类型检查
pnpm lint:eslint    # Vue/TS/TSX 检查与修复
pnpm lint:style     # 样式检查与修复
pnpm build:dev      # 开发环境构建验证
```

推荐最低标准：

- 只改文档：无需构建，可检查 Markdown 格式。
- 改 TS 类型、API、Hook、Store：执行 `pnpm ts:check`。
- 改 Vue 页面或组件：执行 `pnpm ts:check` 和 `pnpm lint:eslint`。
- 改样式：执行 `pnpm lint:style`。
- 改路由、构建、全局组件、权限：执行 `pnpm build:dev`。

## AI 生成代码提示词建议

给 AI 分配任务时，建议包含：

```text
请先阅读 docs/开发强制约束.md、docs/工程化规范.md，以及相关组件文档。
本需求需要遵循现有 Vue3 + Element Plus + Pinia + Vite 架构。
优先复用项目已有 Table/Form/Search/Modal/useTable/useForm/useSearch 封装。
完成后说明改动文件、关键实现、已执行或未执行的检查命令。
```

页面类需求建议补充：

```text
请实现真实可用的中后台页面，不要做营销页。
需要包含查询、列表、分页、加载态、空态、提交 loading、删除确认、表单校验。
接口和字段以现有 API 或需求文档为准，不要自行编造。
```

## 新功能交付清单

- 是否新增或更新 API 类型。
- 是否复用项目封装组件。
- 是否处理加载、空态、错误、禁用、重复提交。
- 是否补充路由、菜单、权限、国际化。
- 是否遵循目录和命名规范。
- 是否执行必要质量检查。
- 是否没有改动无关文件。
